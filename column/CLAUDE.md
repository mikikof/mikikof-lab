## 1. 記事制作の全体フロー

ユーザーから素材と指示を受けたら、**次の順番を厳守**してください。途中を飛ばさないこと。

```
① 素材の読み取り   → ② 構成立案   → ③ スラッグ決定
④ スキャフォールド → ⑤ 本文執筆    → ⑥ 文体チェック
⑦ 参考文献整備    → ⑧ 一覧ページ更新 → ⑨ Git commit & push
```

各工程の詳細は下記セクションを参照。

## 2. 素材の読み取り方(入力ソース別)

### 2-1. PDFファイル

ユーザーが PDF をアップロードした場合、ファイルは `/mnt/user-data/uploads/` に配置されます。

```bash
ls /mnt/user-data/uploads/
```

テキストPDFなら `pdftotext` で抽出、スキャン画像PDFなら OCR が必要です。まず次で確認:

```bash
pdftotext -layout /mnt/user-data/uploads/FILENAME.pdf - | head -100
```

空白ばかり返るなら OCR 対象。Python の `pdfplumber` や `PyPDF2` でテーブル構造が必要な場合は使用可。OCRは `tesseract` を使用し、日本語は `-l jpn` を指定。

### 2-2. URL(Wikipedia・ニュース・ブログ等)

`web_fetch` ツールで本文を取得します。複数URLの場合は1つずつ取得し、それぞれから主要な事実と引用可能なポイントを抜き出す。**URLは必ず参考文献として記事末尾に記録**。

### 2-3. スライド(pptx, Keynote)

pptxは `python-pptx` で本文テキストを抽出。Keynoteは一度 pptx か PDF に書き出してもらう必要があるため、ユーザーにそう案内すること。

```python
from pptx import Presentation
prs = Presentation("/mnt/user-data/uploads/FILE.pptx")
for i, slide in enumerate(prs.slides):
    for shape in slide.shapes:
        if shape.has_text_frame:
            print(f"[Slide {i+1}] {shape.text_frame.text}")
```

### 2-4. 画像・写真(テキスト含む)

画像内のテキストは、まずマルチモーダルで直接読み取り可能かを確認。図表や手書きが混ざる場合は `tesseract -l jpn+eng` で補助的にOCRを併用。図そのものの構造を把握したい場合は、画像を記述的に分析してから構成に生かす。

### 2-5. 手書きテキスト・音声メモ

手書き画像は上記のOCRフローで処理。音声メモ(.m4a/.mp3/.wav)は Whisper 系を使用:

```bash
pip install openai-whisper --break-system-packages
whisper /mnt/user-data/uploads/MEMO.m4a --language ja --model small
```

出力された `.txt` を素材として扱う。

### 2-6. 素材の読み取りで守ること

- **素材を要約しすぎない**。記事の元ネタになる固有名詞・年号・数字は全部温存する
- **鮮度が要る情報(時事・最新動向)**は必ず `web_search` で追加確認
- **一次ソースを優先**。Wikipediaを元にした記事でも、本文で参照している典拠を必ず辿る

## 3. 構成立案

素材を読み終えたら、まず「その記事の最適な章数」を判断します。以下を目安に:

| 題材の性質 | 推奨構成 |
|---|---|
| 単一概念の深掘り(例: 「対数とは何か」) | **3章** + 導入 + 結び |
| 歴史の流れや比較を伴う題材(例: 「大陸法vs英米法」) | **5章** が標準 |
| 多層的・学際的題材(例: 「量子コンピュータと社会」) | **7章**まで拡張可 |
| 専門題材で前提知識の解説が必要(例: 「暗号資産とミームコイン」) | **0章プレリュード + 5章 = 6章** |

章数に関係なく、記事には次のブロックを必ず含めること:

1. **ヒーロー**(タイトル + リード文 + スクロール誘導)
2. **目次セクション(`.toc-section`)**(ヒーロー直後・必須)
3. **デスクトップ固定サイドレール(`.rail`)** + **モバイルドロワー(`.m-drawer`)**(必須)
4. **本編の各章**(章番号・タイトル・英訳・リード・本文 + ビジュアル要素)
5. **参考文献**(記事末尾。後述)
6. **フッター**(記事タイトルと「← 記事一覧へ戻る」リンク)

ヒーロー直下の目次・サイドレール・モバイルドロワーの三点は[`skills/visual-column/SKILL.md`](skills/visual-column/SKILL.md) §3 で詳細を定義しています。0章プレリュードの設計原則は同 §4-2 を参照。

### 章立ての良い例

- 「思想 → 地図 → 歴史 → 実践 → 現代日本の位置」(法の二大系統)
- 「原理 → 仕組み → 応用 → 限界 → 未来」(技術解説)
- 「出来事 → 背景 → 当事者 → 影響 → 教訓」(時事解説)

### ビジュアル要素の選び方(重要)

**必ず1〜3つ入れる**。テキストだけの記事にしない。題材に応じて次から選択:

- **対比ダイアグラム**(2列比較、SVGで図解)
- **世界地図 / 日本地図**(d3.js + world-atlas / geojson を使用、本物の地図のみ)
- **年表 / タイムライン**(縦スパインの左右展開 or 横スクロール)
- **インタラクティブシナリオ**(2択ボタンで視点切替)
- **階層図 / 構造図**(今回の日本法の「二階建て」のようなもの)

地図や年表のような大きな仕掛けは、**データが実在する素材のときだけ**使う。それ以外は対比ダイアグラムや階層図で十分成立する。

## 4. スラッグとファイル配置

記事のスラッグ(URLに使う短縮名)は、次のルールで決定:

- **英小文字 + ハイフン区切り**。例: `civil-vs-common-law`, `entropy-and-life`
- **5単語以内**、40文字以内
- 同名が既存なら末尾に `-2` 等をつける

ファイルは次に配置:

```
column/articles/{slug}/index.html
```

共有CSSやJSは使いません。**各記事のHTMLは自己完結**(インライン `<style>` と `<script>` で書く)。これはメンテ性より「その記事単独でも1ファイルで動く」安定性を優先した判断です。

**モノレポでの作業位置**: スクリプト類はすべて `column/scripts/` 配下にあります。スクリプトは `column/` ディレクトリから実行することを想定しているため、作業開始時は `cd column/` で移動してから操作してください。

## 5. 新記事のスキャフォールド

## 6. デザイン・レイアウトの固定ルール

**指示がない限り、`templates/base.html` のデザインと[`skills/visual-column/SKILL.md`](skills/visual-column/SKILL.md) §2 の固定値を一切崩さないこと**。崩していいのは以下の1点のみです。

### 6-0. レイアウトの基本骨格(2026年4月以降の標準)

- チャプターは `max-width: 920px` の単一カラムを画面中央に配置
- ヘッダ系(キッカー・タイトル・サブタイトル・リード)は中央揃え
- 本文段落は読みやすさのため左揃え、ただしカラム自体は中央寄せ
- ヒーロー直下に `.toc-section`(必須)
- ビューポート ≥1180px で `.rail`(サイドレール)を表示
- ビューポート <1180px で `.m-menu-btn` + `.m-drawer`(モバイルドロワー)
- IntersectionObserver で現在章を `.active` 化(JS は components.md §10 から)

詳細寸法と部品の HTML/CSS は [`skills/visual-column/components.md`](skills/visual-column/components.md) を参照。

### 6-1. 変えていいもの: 配色テーマ

`templates/palettes.md` に定義された科目別パレットから、題材に合うものを選んで CSS 変数を差し替える。例:

- 情報科学・数学系 → **Indigo × Cyan**
- 物理・化学系 → **Navy × Amber**
- 生物・医学系 → **Forest × Coral**
- 歴史・政治系 → **Indigo × Coral**(今回の法サイトの配色)
- 経済・ビジネス → **Charcoal × Gold**
- 哲学・文学系 → **Burgundy × Cream**

### 6-2. 変えてはいけないもの

- **フォント構成**: 見出し `Zen Old Mincho` / 本文 `Zen Kaku Gothic New` / ラテン `Fraunces` + `DM Sans`
- **タイポグラフィ**: 本文 18px / weight 500 / line-height 2.0 / letter-spacing 0.025em
- **レイアウト構造**: ヒーロー → 章(ナンバリング + カテゴリ + タイトル + 英訳 + サブ + リード + 本文) → 章間の区切り方
- **背景**: `#f7fafd` の青みのある白 + 極薄ドットパターン
- **スクロールで隠れるトップナビ**の挙動
- **章番号**: 大文字ローマ数字またはアラビア数字(01, 02...)で Fraunces イタリック

### 6-3. ビジュアル要素の技術制約

- SVGを使う場合は viewBox を指定し、レスポンシブに
- 地図は **d3.js v7 + topojson v3 + world-atlas@2 (countries-110m)** を使用。自作の適当な多角形は禁止
- アニメーションはCSSの `@keyframes` を基本に。重いライブラリは読み込まない
- 全ての外部ライブラリは `cdnjs` または `jsdelivr` のCDNから読み込む

## 7. 文体ルール(最重要)

**AI臭い文章は絶対に出さない**。本節のルールは記事本文だけでなく、**記事タイトル・ヒーローのリード・一覧ページのカード説明・フッター**など、このリポジトリで書くすべての文章に適用します。目指すのは、理路整然としつつ読みやすく、アカデミックな気品のある上品な文体です。

### 7-1. 基本の文体

- **です・ます調**で統一。「〜である」「〜だ」「〜のだ」は原則禁止
- 例外: 引用符の中、過去の事実の修飾節内(例:「当時最新鋭**だった**ドイツ民法典」)のみ許容

### 7-2. AI臭を回避するための禁止表現リスト

次の表現は**使ってはいけません**。見つけ次第、別の表現に置き換えてください。

| 禁止表現 | なぜダメか | 代替案 |
|---|---|---|
| 〜していきます / 〜ていく | 過剰な進行形で冗長 | 〜します / 〜する |
| 〜てきます / 〜てくる | 同上 | 〜します |
| 〜のぞきこむと | 説明口調の定番 | (省く or 具体動詞に) |
| 〜見えてきます / 浮かび上がります | 同上 | 〜が分かる / 〜に気づく |
| 〜というタイプの | 直訳調 | 〜という / 〜のような |
| 〜ということです / 〜というわけです | 曖昧・冗長 | 直接断定 |
| 〜となります / 〜となっていく | 過剰 | 〜になる / 〜です |
| ユニーク / ピッタリ / バッチリ | カタカナ俗語 | 独自の / ぴたりと合う |
| ズバリ / とにかく | 口語すぎ | (省く) |
| まさに / つまり / ほぼ(連発) | 強調の過剰使用 | 1記事1〜2回まで |
| 〜のです(連発) | 説諭調の過剰 | 1段落に1回まで |
| ちゃぶ台返し / ブチ切れ / ベタ褒め | 俗語・過度にくだけた表現 | 「白紙に戻す」「激しく反発」「絶賛」 |
| 〜わけです / 〜というわけ | 翻訳調 | 直接断定 |
| 深掘り / 掘り下げる | 誇張・マーケ調 | (省く) or 「詳しく扱う」 |
| 詰め込む / 満載 / 凝縮 / 余すところなく | 宣伝口調 | (省く) or 「収めている」 |
| 立体化する / 紐解く / 学びのハブ | AI頻出のメタファー | (省く or 具体動詞で書き直す) |
| 別のかたちで / もう一度〜を | 感傷的キャッチコピー | (省く) |
| 〜のための場所 / 〜のための〜 | ポスター調の冠 | (省く or 内容ベース) |
| 〜を、〜に。 / 〜を。(体言止め標語) | AI臭のあるヘッドライン | 普通の文で書く |

### 7-3. 目指す文体の質感

- **漢語とやまと言葉のバランス**: 名詞に過度な漢語を積まず、動詞で切る
- **リズム**: 長文→短文→長文 の呼吸。短文を恐れない
- **アカデミックな語彙**: 「受容する」「系譜」「流儀」「金字塔」「書架を巡る」のような、教養を感じさせる語を適所に
- **体言止めの節度**: 段落末に1回程度。多用禁止
- **具体で書く**: 「〜の視点」「〜の側面」といった抽象語を避け、「条文を開く」「書架を巡る」のように動詞と具体物で書く
- **一文の長さ**: 基本40〜60字。長くても100字超は分割

### 7-4. 書き出しの原則

- ヒーローのリード文は3行構成が基本(1行目で全体像、2行目で対比、3行目で本記事の狙い)
- 各章のリード文は、その章で「何をこれから話すか」を90字以内で提示
- **段落冒頭に「ところで」「さて」を連発しない**(2回まで)

### 7-5. 結びの原則

- 最後の章は「教訓 + 読者への開いた問い」で閉じる
- 「〜するはずです」「〜と言えるでしょう」のような柔らかい断定で終わる
- 「以上です」「まとめると」のようなベタな締めは禁止

### 7-6. 見出し・リード・カード説明文のトーン

記事本文だけでなく、**記事タイトル、ヒーローのリード文、トップページのカード説明、フッターの一行**も同じ基準で書く。「AIが書いたポスター」にならないための追加ルール:

- **標語調を禁止**。「授業の現場で、もう一度解ける教材を。」のような「〜を。」「〜に。」で止める体言止め標語は使わない。普通の文で書く
- **抽象メタファーを禁止**。「立体化する」「別のかたちで学び直す」「紐解く」「学びのハブ」はどれもAI臭の発生源
- **対象属性の明示をしない**。「受験生のための〜」「大学生向け〜」「進学校の生徒向け〜」のような冠はつけず、内容ベースで書く(例:「情報Ⅰの各単元を〜」)
- **誇張形容を削る**。「深掘り」「詰め込む」「満載」「凝縮」「余すところなく」は使わない
- **事実の列挙で十分**。カード説明文は「何を・どのように・何のために」が1〜2文で伝われば足りる。形容よりも名詞、比喩よりも動詞

#### Before / After 例(実際に直した事例)

| ✗ AI臭い | ✓ 直したあと |
|---|---|
| 受験生のための、学習ツール実験室 | 学習用ツール置き場 |
| 教科書では分断されている知識を、別のかたちで学び直すための場所。ビジュアル・エッセイ、一問一答、さまざまな角度から、あなたの学びを立体化します。 | エッセイ、講義用資料などをまとめています。 |
| 授業の現場で、もう一度解ける教材を。 | 情報Ⅰの授業用インタラクティブ教材 |
| 情報Ⅰをはじめとする授業単元を、大学生・進学校の生徒向けに、投影用スライドと復習用Webサイトの両方で機能する形に。クリック展開式の穴埋め、リアルタイム計算機、実例の深掘りまで、単一HTMLに詰め込んでいます。 | 各単元は、授業中の投影と授業後の復習の両方に使えます。穴埋めや計算機などは、クリックで答えが展開する形式です。 |

目指しているのは、**理路整然としつつ読みやすく、アカデミックな気品のある**文章。硬すぎず、軽すぎず、宣伝文句にもならない。大学生の教養読み物として成立するトーンを、見出しやカード1枚の文言にまで貫くこと。

## 8. 引用・出典のルール

**全ての記事は、末尾に参考文献セクションを必ず設ける**。例外なし。

### 8-1. 参考文献の書式

記事末尾、フッターの直前に配置:

```html
<section class="refs">
  <div class="refs-head">参考文献</div>
  <ol class="refs-list">
    <li>
      <span class="ref-author">著者名</span>
      <span class="ref-title">『書名』または「記事タイトル」</span>
      <span class="ref-meta">出版社, 年 / URL</span>
    </li>
    <!-- ... -->
  </ol>
  <div class="refs-note">
    本記事は上記ソースを元に、受験生向けに編集・再構成したものです。
    正確な記述については原典をご参照ください。
  </div>
</section>
```

### 8-2. 引用元の選び方

- **一次ソースを優先**。法律なら原文、歴史なら学術書、科学なら査読論文
- Wikipediaは参考にしてよいが、出典としては**その先の原典**を明記する
- URL出典は `web_fetch` で実在を確認したものだけ記載(404を書かない)
- 国立国会図書館・学術機関・政府機関のURLを優先

### 8-3. 本文中の引用

本文中で直接引用する場合は `<blockquote class="cite">` を使い、その引用が参考文献リストの何番目に対応するか `data-ref="N"` で示す:

```html
<blockquote class="cite" data-ref="1">
  過失によって他人の権利を侵害した者は、損害賠償の責を負う。
  <footer>— 日本民法第709条</footer>
</blockquote>
```

## 9. 公開前チェックリスト

**git commit する前に、以下を必ず実行**。スクリプト化されているので1行で回せます:

```bash
bash scripts/check.sh articles/{slug}/index.html
```

手動でも確認するポイント:

- [ ] HTMLタグのバランス(開き/閉じ)が取れている
- [ ] `<script>` と `<style>` の中括弧・カッコがバランスしている
- [ ] 禁止表現リスト(§7-2)に該当する語が含まれていない
- [ ] 全ての本文末尾が「です・ます」で終わっている(修飾節内の「〜だった」等を除く)
- [ ] 参考文献セクションが存在し、最低2件以上の出典がある
- [ ] トップページ `index.html` に新記事へのリンクが追加されている
- [ ] CSS変数(配色)がパレットから正しく選ばれている
- [ ] モバイル表示(幅600px)でも破綻しない(メディアクエリ確認)
- [ ] 記事タイトルが `<title>` タグと `<h1>` で一致している

ナビゲーション系の追加チェックリストは [`skills/visual-column/SKILL.md`](skills/visual-column/SKILL.md) §7 にあります。デスクトップではサイドレールがスクロールに追従するか、モバイルではドロワーが下から開いて自動で閉じるかを、必ず実機で確認してください。

## 10. トップページ `index.html` の更新

新記事を公開したら、必ず `index.html` の記事カードリストに新しいカードを追加してください。カード1枚のHTMLテンプレは `index.html` のコメント内に記載してあります。

記事カードに必要な情報:

- 記事タイトル(日本語)
- 英訳サブタイトル
- 科目カテゴリ(情報 / 歴史 / 経済 etc.)
- 公開日(ISO形式: `2026-04-21`)
- 1〜2行の要約
- 記事ページへのリンク

最新記事が上に来るように、カードの順序は新しい順。

## 11. Git commit & push(自動化)

認証は**既にローカルで済ませてある前提**で、以下のスクリプトを実行するだけで push されます:

```bash
# column/ ディレクトリから実行すること
cd column/
bash scripts/publish.sh "civil-vs-common-law" "column: 法の二大系統を公開"
```

スクリプトは内部で、モノレポ全体のgitリポジトリ(`mikikof-lab/`)に対して以下を実行します:
1. `git add column/articles/{slug}/ column/index.html`
2. `git commit -m "{メッセージ}"`
3. `git push origin main`

**コミットメッセージの prefix ルール**: 本ツールのコミットは必ず `column:` で始める。例:
- `column: 法の二大系統の記事を追加`
- `column: 量子コンピュータ解説の図解を修正`
- `column: 配色テーマに Ocean × Sunset を追加`

これにより、将来 `quiz/` や他ツールが増えても、コミット履歴を見て「どのツールの変更か」が一目で分かります。

## 12. 守ってほしい「いい記事」の条件

CLAUDE.md のここまでのルールを守った上で、**さらに一歩踏み込んで**ほしいこと:

1. **題材に愛を持って書く**。素材を右から左に流すのではなく、「これはなぜ面白いのか」をまず自分で整理してから書き始める
2. **世界史・日本史の固有名詞を積極的に本文に織り込む**。受験生はそれで世界史の知識と繋げて覚えられる
3. **専門用語は、初出時に必ず1行でかみ砕く**。専門用語そのものは削らない。レベルを落とさないことがアカデミック
4. **図と文のバランス**。図が多すぎても少なすぎてもダメ。3章なら1〜2図、5章なら2〜4図が目安
5. **最後にちょっとした「余韻」を残す**。受験勉強を越えた、読み物としての価値を意識する

## 13. 困ったときの判断指針

- **章数を何章にするか迷ったら** → 素材の密度に応じて3/5/7から選ぶ。短すぎるより短すぎない方が良い
- **ビジュアル要素を入れるか迷ったら** → テキストだけで十分伝わる場合は入れない。無理やり図を作らない
- **禁止表現リストにない言い回しだけどAI臭い気がする** → 疑問に思った時点でアウト。具体動詞と短文で書き直す
- **配色テーマが題材に合わないと感じたら** → `palettes.md` に新パレットを追加してよい。ただし必ず1記事1カラーテーマ
- **参考文献が見つからない** → 記事として成立しない。素材をユーザーに追加で要求する
