# lectures/ — Claude Code 運用指示書(Interactive Lecture Lab)

このフォルダは、大学・高校の授業で使うインタラクティブなHTMLスライドサイトを量産する場所です。**単一HTMLファイル**で投影と復習の両方に使える教材を作ります。

ルートの `CLAUDE.md` からここに到達したら、本ファイルの指示に従って作業してください。

---

## 1. このツールの位置づけ

### ユーザー像
- **作り手**:幹生(平成国際大学 情報デザイン学部 教員)
- **投影者**:授業担当者
- **閲覧者**:受講学生(大学生、および情報Ⅰ共通テスト対策を受ける進学校の生徒)

### 成果物の特徴
- 単一 HTML ファイル(CSS・JS を全て内包、オフライン閲覧可)
- 授業中はプロジェクタ投影、授業後は生徒の自習復習に使える
- 学習ノートの項番(①②③…)と完全対応
- 最新の事例・ニュースを取り込んだ生きた教材
- **同一URLでPC(投影)とスマホ(復習)の両方に最適化**(ユニバーサル対応)
- **開閉式スライド目次ドロワー** + **インタラクション一括リセット** を標準装備

### 公開URL
- 授業サイトトップ:`https://mikikof.github.io/mikikof-lab/lectures/`
- 単元ページ:`https://mikikof.github.io/mikikof-lab/lectures/articles/{slug}/`

## 3. 作業着手前に必ず読むもの

新しい単元を作る・既存を修正するときは、**以下の順で必ず読んでから**コードに触る:

1. 本ファイル(このCLAUDE.md)— 制作哲学とチェックリスト
2. `skills/interactive-lecture/SKILL.md` — 技術仕様と落とし穴
3. `skills/interactive-lecture/components.md` — 部品カタログ
4. `skills/interactive-lecture/examples/` の最新完成品 — 生きた参照
5. **`_source/` の該当章 Word + 解答PDF の対応ページ** — 問題と答えの正本。これを読まずに書くと、答えの誤りや原本と異なる根拠フレームを書く事故が起きる(§4.8 参照)

これを読まずに書き始めると、毎回デザイン規約が揺れて量産の意味がなくなる。**skill と原本を読むのは念のためではなく必須**。

## 4. 制作哲学(これを外したらやり直し)

### 4.1 答えは必ず後出し
- 穴埋め、クイズ、実習問題の正解を先に見せない
- クリック・タップで展開する仕組みに統一する
- 「?」「クリックで確認」などのプレースホルダ表示を使う
- これを破ると**教材ではなく単なる解説資料になる**

### 4.2 問題は手応えのある難度
- 「使い回しはNG」「共用PCはNG」といった自明の問題は出さない
- 引っかけ、グレー判断、複数要素の組み合わせで判断を要する問題にする
- 対象は**共通テスト〜難関大受験生レベル**
- 「これなら教科書見れば分かる」問題は価値がない

### 4.3 実例を入れる
- 抽象的な説明だけでなく、現実の事件・ニュースで具体化する
- 情報セキュリティ系は **web_search で最新事例を確認**してから書く
- 事例は展開式で格納し、スライドを情報過多にしない

### 4.4 子供っぽい表現を避ける
- ❌「やってみよう」「頑張ろう」「みんなで考えてみよう」
- ❌ 過度な絵文字(💡 など機能的記号を除く)
- ✅ 中立的・プロフェッショナルな表現

### 4.5 対象属性の明示をしない
- 「東大受験生向け」等の特定属性明示はしない
- 「このサイトで扱う内容」「共通テストで問われる論点」のような内容ベース記述にする

### 4.6 学習ノートとの対応を明示
- 各スライドのラベルに「学習ノート ①」「実習 2」など対応箇所を表示
- 項番まで揃えることで、学生が手元のノートと照合しながら進められる

### 4.7 見出し・リード文のトーン
- スライド本文だけでなく、**`lectures/index.html` のヒーロー・単元カード説明・フッター**も同じ基準で書く
- ❌ 標語調:「授業の現場で、もう一度解ける教材を。」のような「〜を。」「〜に。」で止める体言止めヘッドライン
- ❌ 抽象メタファー:「立体化」「紐解く」「別のかたちで」「学びのハブ」
- ❌ 誇張形容:「深掘り」「詰め込む」「満載」「凝縮」「余すところなく」
- ❌ 対象属性の冠:「大学生・進学校の生徒向け」(4.5 と同方針)
- ✅ 事実ベースの平易な一文で、単元構成や使い方を淡々と述べる
- 詳細な禁止表現と Before/After はルート `CLAUDE.md` の「文章トーン」節と `column/CLAUDE.md` §7-6 を参照

### 4.8 答えと根拠は原本(解答PDF)を正本として扱う
- lecture に載せる**正答・選択肢の○×・解説の根拠**はすべて `_source/高校情1学習ノート-解答PDF.pdf` の対応箇所を **直接 Read で開いて確認**する
- 「自分の知識でこうなるはず」「論理的にこうだろう」で書かない。原本の解説で使われている**根拠フレーム(条文・概念・論じ方)** をそのまま採用する
- 過去事故:著作権の例外規定問題で、原本は「著作権法35条(学校教育における複製)」フレームで判断していたのに、lecture は「私的使用」「引用」フレームで独自に判断。結論まで誤り、生徒から指摘を受けた(04-IP オの応援旗:正解「適切」を「不適切」と誤判定)
- 独自に4択や応用問題を追加する場合も、**原本の根拠フレームと矛盾しない**ことを確認すること
- agent(Explore など)に答えの監査を委譲した場合、「全件 ✓」報告を鵜呑みにせず、最低でも**指摘箇所と疑問の残る箇所は自分の目で原本PDFを Read して再確認**する

### 4.9 学習ノート POINT 項目ごとに専用ビジュアルを必須化(2026-04 改訂)

学習ノートの各 POINT 項目(①②③… 全項目)には、**用語定義(def-box)に加えて 2 種類のビジュアル要素を必ず付ける**:

1. **law-illust(def-box 上部の SVG イラスト)** — 用語の意味を視覚的に伝えるアイコン or ポンチ図。インライン SVG で `viewBox="0 0 100 100"` の 92×92 px 表示。`components.md` の law-illust パターン参照
2. **インタラクティブ図解** — 関連する概念構造を「タップで反応する図」として実装(三角形・家系図・対比図・フロー図など)。タップ → 解説テキストが下部 info パネルに表示される動作が標準

これを外すと、テキストの羅列だけになり「ビジュアル面が弱い・薄い」教材になる(過去フィードバック参照)。lec06、lec07 が完成形のリファレンス。

**対応コンポーネントは以下から選択(`components.md` 18〜23 参照):**

| 学習項目の性質 | 推奨ビジュアル |
|---|---|
| 定義・概念単発(各 POINT 共通) | **law-illust**(def-box header) |
| 主体間の関係(N者関係) | **interactive-svg-node**(CIA三角・PF三角・青少年保護) |
| 二項対比(A vs B) | **comparison-visual**(紙 vs 電磁的・黒 vs 白リスト) |
| 段階的処理(順序) | **flow-diagram**(ログイン・オプトイン・クーリングオフ) |
| 階層・分類 | **tree-diagram**(マルウェア家系図・法のピラミッド) |
| 判定型クイズ | **judgment-chip**(○✕タップ判定+リセット可) |

**SVG 文字サイズの最小ルール:**
- viewBox `0 0 100 100`、表示 92×92 px の場合、文字は `font-size 9` 以上(理想は 10〜18)
- 円や矩形内の漢字 1 文字は `font-size 11+`(`font-weight 900` 推奨)
- 5px 以下の文字は禁止(モバイル 80×80 px 表示で完全に潰れる)
- 過去事故:lec07 ① CIA 盾の `font-size 7-8` 文字が「小さすぎて読めない」と指摘された

**SVG ビジュアル品質ルール(2026-04 lec03 / lec02 で確立):**
- 法則 SVG は **flat polygon + 直接 text** だと即チープ化する。`<defs>` に **`linearGradient` + drop-shadow filter** を仕込み、上端 2-3px のホワイトハイライトで立体感を出す(SKILL.md §9.17)
- テキストは可能な限り SVG 外(`law-illust-tag` / `def-desc`)に逃がす。SVG 内のテキストは「GAP」「?」のような **超短いシンボル** に限定し、必ず塗りのある rect / circle / polygon の上に置く(canvas 余白に置くと白に同化)
- `law-illust-tag` は **1 単語**(`DEDUCTIVE` / `TRADE-OFF` 程度)に短縮。letter-spacing 0.2em と相まって 92px 幅で切れる

**SVG 統一感ルール(2026-04 lec02 で確立 / SKILL.md §9.18 §13):**
- 単元内の **すべての SVG は同じ design system** で統一。バラバラだと「素人くさい」見え方になる
- 共通の `<defs>`: `icoNavy`(linearGradient 5d8eb3→0F2847)/ `icoGold`(linearGradient f4d995→8a6d1f)/ `icoBg`(radialGradient 白→F3F6FB)/ `icoSh`(drop-shadow filter)を全 SVG で使い回す
- **役割を色で固定**: navy グラデ = 主役、gold グラデ/単色 = アクセント・正解、red グラデ/単色 = 対比・警告、navy-pale + opacity = 補助線
- シルエット重視。「目を細めて見ても何か分かる」レベルのアウトラインを最初に決めてから詳細を足す
- 自作で揃えるのが難しい場合、**Lucide / Phosphor / Heroicons / Tabler / Material Symbols** などの **OSS アイコン ライブラリ**(MIT/CC0/Apache)から SVG path を借りる。**1 単元 = 1 ライブラリ** が原則(混ぜない)。詳細は SKILL.md §13
- 単元の全 SVG を並べて見比べる:「同じデザイナーが描いた」と感じなければ揃え直す。canonical 参照は `articles/02-society-and-it/`(9 法則 SVG が design system で統一)

### 4.11 SVG 内テキストでラベルを書かない、HTML 要素として配置(2026-05 lec01 制作で確立)

- 「② POINT」「残存性」「⑤ 生命情報」のような **役割テキスト・タグ・名称ラベル**を SVG 内 `<text>` で書くと、円・楕円のリング線や Phosphor アイコンと重なって読めなくなる(lec01 制作時にユーザーから「文字が被って可読性が低い」と指摘)
- **新ルール**: SVG は「アイコン本体だけ」を担う。タグ(POINT 番号)・名称(残存性 等)・サブ説明はすべて **HTML 要素**(`.icon-card-tag` / `.icon-card-name` / `.icon-card-sub`)としてレンダリングする
- インタラクション(タップで詳細パネル表示)も `.icon-card` の `onclick` で取れる。SVG 全体を 1 つの `<g class="prop-circle">` として扱う必要はなくなった
- canonical 参照: `articles/01-information-and-media/index.html` の SLIDE 5(情報の3特性 prop-card)/ SLIDE 12(メディア3分類 med-card)
- 例外: 楕円の入れ子構造のように **入れ子の各レイヤを示すラベル** は SVG 内に残してよい(SLIDE 8 nest-svg)。ただしリング線から十分離す(font-size 13+、リングから 15px 以上)
- 詳細実装は `skills/interactive-lecture/SKILL.md` §14 icon-card パターン参照

### 4.12 SVG 背景は透過、`<rect width fill>` は禁止(2026-05 lec01 制作で確立)

- 各 SVG の冒頭に `<rect width="256" height="256" fill="url(#iB1)"/>` のような背景矩形を置くと、コンテナ(`.def-box` / `.law-illust` 等)の中に **「白い四角」が浮いて見える**(lec01 制作時に指摘)
- **新ルール**: 全 SVG は **完全透過**。背景の薄ゴールド帯・薄水色は **コンテナ側の CSS** で表現する
- `.law-illust` の CSS background も `transparent` を基本に。アクセントが欲しければ `border-bottom: 1px dashed var(--navy-pale)` のような細い線で代替
- 詳細は `SKILL.md` §9.19

### 4.13 学習ノートが浅い単元は「深掘り副題」セクションを追加する(2026-05 lec01 制作で確立)

- 学習ノートの POINT・実習だけだと、共通テスト・大学入試・実生活で必要な射程に届かないことがある(lec01 では印象操作の実習が 3 問だけ → ユーザーから「もっとページ数を割いて深堀したい」と指摘)
- 該当する単元では、学習ノートの末尾(または該当セクション末尾)に **1〜4 スライドの深掘り副題** を追加する

| 元の単元 | 深掘り副題候補 |
|---|---|
| 情報の定義(lec01 ①) | DIKW 階層(データ→情報→知識→知恵) |
| 印象操作・統計(lec01 実習5) | グラフ基本 6 型 / 印象操作 9 技法(目盛り/視覚効果/比較の罠) |
| 知的財産(lec04) | 著作権の歴史的経緯 / Creative Commons 4 種ライセンス |
| 個人情報(lec05) | 個人情報保護法の改正史 / GDPR 比較 |
| 情報法(lec06) | 法のピラミッド(憲法→法律→政令→省令) |
| セキュリティ(lec07) | CIA トライアド詳解 / 最新インシデント事例 |
| 問題発見と解決(lec03) | PPDAC / 演繹と帰納の厳密化(§4.14 と連動) |

- 判断基準: **共通テスト・大学入試・実生活**で問われる射程を持つか。学習指導要領に明記されていなくても、教材として薄ければ補強
- 「もっとページ数を」と言われそうな広がりがあるなら、最初から深掘りスライドを入れておく
- 副題セクションは `<h3 class="section-heading">参考: ...</h3>` で始め、本来の学習ノート単元と区別する

### 4.14 概念定義の断定を避ける(2026-04 lec03 codex audit より)

教材は「正解は X」と言い切りやすいが、**論理学・法学・社会科学の抽象概念** は、断定するほど厳密性が崩れる。以下のような場面では、断定の前に専門観点で締め直す習慣を持つ:

- **論理学**: 「演繹は確実」「妥当性が保証される」 → **「妥当な推論形式なら」が前提**(健全性 = 推論形式の妥当性 + 前提の真実性)
- **論理学**: 「帰納の結論は反例で崩れる」 → **全称命題と統計的仮説で扱いが違う**(全称命題は反例 1 つで否定、統計的仮説は信頼度低下に留まる)
- **法学**: 「著作権 32 条の引用要件は ① ② ③ ④ で、これを満たせば許諾なく引用できる」 → **条文中核は「公正な慣行」「引用の目的上正当な範囲」**。4 要件は実務的目安で、断定すると条文解釈を歪める。出所明示は **48 条** 由来として併記
- **探究学習**: 「課題=具体的な行動・方策」 → **問題 → 課題(問いの形)→ 解決策(具体施策)** の 3 層を分ける。「課題」と「解決策」を混ぜると階層構造が壊れる

過去事例: lec03 (2026-04-30 codex audit) で上記 4 点を指摘 → 修正。詳細は `.company/audit/reviews/2026-04-30/2138-lec03-problem-solving/` 参照。

**対策**: 概念定義のスライドを書くとき、その分野の専門家(法学者・論理学者・統計学者)が見ても破綻しない表現か自問する。教科書(高校レベル)に書いてある通りでも、それが大学レベルで通用するか確認。迷ったら `/audit-review` で外部レビューを取る(§6 [6.5] 参照)。

## 5. 技術的な制約

### 必須
- **単一HTMLファイル**で完結
- CSS・JS は全て内包
- Google Fonts の読み込みのみ外部 CDN として許可
- Safari(macOS / iPad)と Chrome で動作

### 禁止
- ❌ フレームワーク(React, Vue, Alpine.js 等)の導入
- ❌ localStorage / sessionStorage の使用
- ❌ 重いライブラリの CDN 読み込み
- ❌ ビルドステップが必要な構成(そのまま GitHub Pages で配信できない)

### ナビゲーション(標準装備)
- 矢印キー(← →)、スペース、PageUp/Down、Home/End
- タッチスワイプ(iPad / iPhone / Android)— 水平 40px 以上・800ms 以内で反応
- F キーで全画面切替
- M キー(または topbar 右端の `MENU` ボタン)で**スライド目次ドロワー**を開閉
- 進捗バー + ページ番号常時表示
- ESC でドロワーを閉じる

### スライド目次ドロワー(標準装備)
- セクション(SECTION 1〜N)ごとに自動でグルーピング
- 各項目に `PRACTICE` / `INTERACTIVE` / `REVIEW` などのタグを自動付与
- 現在スライドをハイライト、タップで任意のスライドへジャンプ
- 上部に **「インタラクション全リセット」** ボタン

### ユニバーサル対応(PC + スマホ)
- 同一URLで **PC(1100px+):横長スライド** / **タブレット(720–1100px):簡略化** / **スマホ(〜720px):縦に最適化された復習モード** を自動切り替え
- iPhone のセーフエリア(ホームインジケータ)に食い込まない
- 新規コンポーネントを追加するときは、必ず `@media (max-width: 720px)` も書く

詳細仕様は `skills/interactive-lecture/SKILL.md` を参照。

## 6. 標準ワークフロー(新規単元1本)

新しい授業単元の依頼を受けたら、以下の順で進める:

### [5] 答えの照合(原本との突き合わせ) ★ 飛ばすと事故る
- lecture 内のすべての問題について、`_source/高校情1学習ノート-解答PDF.pdf` の対応箇所と以下を1問ずつ照合:
  - **正答(○×・選択記号)** が一致しているか
  - **解説の根拠フレーム**(条文・概念・論じ方)が原本と揃っているか — 結論だけ合っていても、根拠が原本と異なると別の問題で破綻する(§4.8 参照)
- 独自に追加した4択クイズや応用問題は、原本の根拠フレームと矛盾しないことを確認
- このステップの結果はユーザーへの報告に必ず含める(「解答PDF p.X〜Y と全問照合済み」のように)
